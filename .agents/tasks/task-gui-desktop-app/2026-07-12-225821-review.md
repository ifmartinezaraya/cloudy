# Desktop GUI shell for CloudVault (CustomTkinter + pystray)

A full Python GUI layer wraps the existing PowerShell management scripts. Six views (Dashboard, Sync, Clouds, Settings, Logs, Restore) hang off a sidebar in a single-window CustomTkinter app. A system tray icon provides background persistence. The whole thing is themed purple (#8B5CF6) on jet black (#0a0a0a), with a programmatically generated cloud .ico for window and tray icons. A PyInstaller build script packages it all as a standalone .exe.

Watch for:
- **Window close handler typo** (`WM_DELETE_CLOSE` instead of `WM_DELETE_WINDOW`) means minimize-to-tray never fires and close-button does nothing (confirmed).
- **Subprocess command injection surface** in `ScriptRunner._build_command` — user-controlled setting values are passed as bare string arguments to `powershell.exe` without quoting or sanitization (confirmed).
- **No input validation** on Settings save — arbitrary strings written into `settings.json` could corrupt configuration or be injected into PowerShell parameters on next invocation (confirmed).
- **StatusMonitor callbacks fire from a background thread** with no thread-safety for the callback list mutations (`add_callback` / `remove_callback` race with iteration in `_do_check`) (confirmed).

## High-level view

The application is a classically-structured CustomTkinter monolith: `CloudVaultApp` owns a sidebar, a content area, and the core singletons (`ScriptRunner`, `SettingsManager`, `StatusMonitor`, `TrayIcon`). Views are instantiated eagerly at startup and swapped via `grid`/`grid_forget`. The sidebar uses unicode glyphs as icons to avoid asset dependencies.

Script execution funnels through `ScriptRunner`, which builds `powershell.exe -File ...` commands with parameter maps translated to `-Name Value` pairs. Any callback-bearing call runs in a daemon thread and returns results asynchronously. The critical gap here is the absence of shell-safe quoting for parameter values, which exposes the system to argument injection when values originate from the settings UI.

`StatusMonitor` polls `status.ps1 -Json` on a configurable interval (default: health_check_interval * 60 seconds) and pushes `SystemHealth` snapshots to registered callbacks. The callbacks are invoked from the monitor's background thread; each view is responsible for marshalling to the main thread via `self.after(0, ...)`. The callback list itself is not protected by the lock, creating a race between registration and notification.

The system tray integration gracefully degrades: if `pystray` or `Pillow` are absent the app still starts. However, the window-close protocol is registered with the wrong event name, so the minimize-to-tray path is unreachable in practice.

Settings management reads and writes `config/settings.json` with deep-key dot-notation access. The settings view collects freeform text from Entry widgets, attempts an int/float parse, and dumps results back to JSON. There's no schema validation or range checking, so the user can easily write values that break downstream scripts or the monitor interval calculation.

## Window protocol typo

In `gui/app.py`:

```python
self.protocol("WM_DELETE_CLOSE", self._on_close)
```

The correct Tkinter event is `"WM_DELETE_WINDOW"`. The string `WM_DELETE_CLOSE` is silently ignored by Tk, so clicking the window's X button triggers the default destroy behavior — bypassing `_on_close` entirely. The tray minimize logic and the "app still running" notification are dead code until this is fixed.

## Argument injection through ScriptRunner

`_build_command` constructs PowerShell invocations like:

```python
cmd.append(f"-{key}")
cmd.append(str(value))
```

Values come from `SettingsManager` properties, which in turn come from the settings UI or the JSON file. If a user (or a corrupted JSON) places a value like `"; Remove-Item C:\ -Recurse"` in the bandwidth limit field, it's passed as a bare argument. Because `subprocess.Popen` with a list on Windows still goes through `CreateProcess` with space-joined args under certain conditions, and because PowerShell interprets `;` as a statement separator in some calling conventions, this is a real injection surface when `shell=True` is used or when the value happens to be interpreted by PowerShell's parameter binder.

More practically: values with spaces, semicolons, or special characters will at minimum cause the script to fail with confusing errors. At worst, they allow arbitrary command execution. The fix is to validate all values at the settings layer and/or quote them before building the command.

## StatusMonitor thread-safety gap

```python
def _do_check(self):
    ...
    for callback in self._callbacks:  # iterated on monitor thread
        try:
            callback(self._health)
        except Exception:
            pass
```

Meanwhile, `add_callback` and `remove_callback` mutate the same list from the main thread with no synchronization. A `RuntimeError: list changed size during iteration` is possible if a view registers or unregisters a callback during a health check cycle. The `_lock` already exists but only protects `_health`. Extending it to cover the callback list (or using a copy-on-iterate pattern) would fix this.

## Sync cancel does not kill the process

`SyncView._cancel_sync` sets UI state back to idle and logs a message, but the actual sync subprocess is running inside `ScriptRunner.run_async` in a daemon thread. There's no mechanism to store the `Popen` handle and call `.kill()` on it. The sync will run to completion (or timeout after 300 seconds) regardless of the cancel button. The user sees a reset UI while the subprocess continues consuming bandwidth and disk I/O in the background.

## Interactive script invocation

`configure-cloud.ps1` is designed to prompt the user interactively (select provider, enter API keys, etc.). `ScriptRunner` passes `-NonInteractive` and captures stdout/stderr via PIPE, so the script will either hang waiting for input that never arrives or exit immediately. The "Reconfigurar" button in `CloudsView` and the "Configurar Nuevos" header button both call this path. The correct approach would be to spawn a visible PowerShell terminal (`subprocess.Popen` without PIPE capture and without `CREATE_NO_WINDOW`) or build an equivalent configuration flow directly in the GUI.

## Icon generation and asset lifecycle

The icon generator (`generate_icon.py`) uses Pillow's ellipse primitives to compose a cloud shape and saves multi-resolution .ico and .png files. The `.gitignore` excludes the generated `cloud.ico`, and the build script generates it on-demand if missing. The window icon load in `app.py` silently falls through if the file doesn't exist, so a fresh clone without running the generator will produce a window with no icon — functional but visually inconsistent with the design intent.

## Theme architecture

`Theme` is a pure class of constants with no instances — all access is via class attributes. This is clean for a single-theme app. The `get_ctk_theme()` method returns a dict formatted for CustomTkinter's theme system but is never called anywhere in the codebase. It appears to be dead code, possibly intended for a future dynamic theme injection that was never wired up.
